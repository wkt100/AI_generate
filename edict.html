Edict 多智能体协同优化方案

针对 Edict 项目中“三省六部”多 Agent 架构在执行复杂任务时出现的状态错误、循环死锁、上下文遗忘等问题，本方案提供从全局架构、状态机模型到单节点 Prompt 工程的深度优化与重构建议。

一、 现状诊断与错误归因

本区域分析了当前 Edict 项目中“六部” Agent 无法完成任务的核心原因。通过对典型失败案例的解构，我们发现问题并非单点能力不足，而是缺乏全局的约束与规范的数据流转。下面的图表展示了导致任务失败的各类错误的频率。

⚠️核心痛点 1：状态幻觉与越权

各个“部”在执行时，往往忘记当前所处的具体步骤，导致重复执行或越权执行其他部门的职能。例如“工部”直接跳过“中书省”去解析原始需求。

🔄核心痛点 2：缺少有限状态机 (FSM) 强约束

由于使用自由文本进行 Agent 间通信，缺少结构化的 JSON Schema 校验，导致输入输出格式错乱，下游 Agent 无法解析上游结果，从而引发系统崩溃。

🧠核心痛点 3：全局上下文污染

所有历史对话被无差别地灌入每个 Agent 的上下文中，导致 Token 超载和注意力分散。“六部”无法精准提取与自身当前任务相关的数据。

优化要点 1：结构化输出 (Structured Output) 禁止 Agent 使用 Markdown 或自由文本通信。强制使用 Pydantic 模型约束输出为强类型 JSON。例如，如果 `PLAN` 状态不输出 `{"tasks": [...], "dependencies": {...}}`，系统引擎直接拦截并抛出错误重试，而不是传给下游。
优化要点 2：隔离上下文 (Context Isolation) 只传递当前状态所需的数据。执行层（六部）的输入只包含“当前子任务目标”和“上一步局部结果”，绝不包含整个项目的原始聊天记录。减少上下文污染。
